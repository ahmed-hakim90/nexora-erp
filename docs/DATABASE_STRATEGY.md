# Database Strategy

## Database Platform

Nexora uses PostgreSQL through Supabase.

PostgreSQL is the system of record for business data. Supabase provides authentication integration, managed Postgres, Row Level Security, storage, realtime capabilities where appropriate, and operational tooling.

The database must be designed as an enterprise operational database, not as a collection of screen-specific tables.

## Core Data Principles

All database design must follow these principles:

- Tenant isolation is mandatory.
- Business documents must be immutable where legally or financially required.
- Statuses must be controlled by registries and workflow rules.
- IDs must not leak business meaning.
- Document numbers must be generated by numbering services.
- Audit-sensitive changes must be traceable.
- Heavy reporting must not degrade transactional workflows.
- Indexes must reflect real access patterns.
- Foreign keys must protect data integrity.
- Soft delete must be deliberate, not automatic everywhere.
- Scoped uniqueness must include tenant, company, or branch dimensions where relevant.
- Financial and inventory ledgers must be append-focused and reversible by correction entries, not edited in place.
- Schema design must consider RLS performance before tables reach production volume.

## Tenancy Model

The required tenancy concepts are:

- `tenants`: top-level isolation boundary.
- `companies`: legal or operating entities under tenants.
- `branches`: operational locations under companies.
- `users`: authenticated identities.
- `employees`: HR identities, linked to users when applicable.

Rules:

- Every tenant-owned table must include `tenant_id`.
- Every company-owned table must include `company_id`.
- Branch-scoped operational tables must include `branch_id`.
- Cross-branch documents must include explicit source and destination scopes.
- `tenant_id`, `company_id`, and `branch_id` must be sourced from authenticated context and explicit user selection, not hardcoded constants.

Tenant-owned rows must never rely on nullable `tenant_id` as an access shortcut. If a table is global reference data, it must be explicitly classified as global and must not contain tenant-private data.

## Sprint 2 Core Schema

Sprint 2 creates only the minimum platform database foundation in `supabase/migrations`.

Created core tables:

- `profiles`
- `tenants`
- `branches`
- `tenant_memberships`
- `roles`
- `permissions`
- `role_permissions`
- `user_roles`
- `audit_logs`
- `feature_flags`
- `app_settings`

No business module tables are created in Sprint 2.

For Sprint 2, `tenants` represents the top-level organization/company boundary needed for membership, RBAC, and RLS. A separate `companies` table is deferred until multi-company operations are implemented. Future company tables must preserve the existing `tenant_id` isolation boundary.

Role seed data is implemented as system role templates inside `roles` with `role_scope = 'template'` and `tenant_id = null`. Tenant-specific roles remain scoped with `role_scope = 'tenant'` and a non-null `tenant_id`.

## Sprint 3 Platform Engine Schema

Created engine tables:

- `workflow_definitions`
- `workflow_transition_history`
- `approval_policies`
- `approval_instances`
- `approval_steps`
- `approval_history`
- `platform_documents`
- `platform_document_comments`
- `notification_templates`
- `notification_outbox`
- `file_attachments`

These tables are not business modules. They store reusable platform metadata, histories, outbox items, and job state for future modules.

## Scope And Uniqueness Rules

Unique constraints must match business ownership.

Examples:

- Product SKU uniqueness may be `(tenant_id, company_id, sku)` or `(tenant_id, sku)` depending on product strategy.
- Invoice number uniqueness must include the numbering dimensions such as tenant, company, branch, fiscal year, and document type.
- Employee numbers must be scoped by tenant or company according to HR policy.
- Branch codes must be unique within company or tenant, not globally unless required.

No business key should be globally unique by accident if tenants are expected to use the same codes.

## Table Categories

### Platform Tables

Platform tables support capabilities such as:

- Tenants
- Companies
- Branches
- Users and profiles
- Roles
- Permissions
- User assignments
- Feature flags
- Workflows
- Approvals
- Notifications
- Audit logs
- Document numbering
- File attachments

### Master Data Tables

Master data includes relatively stable business entities:

- Customers
- Suppliers
- Products
- Warehouses
- Chart of accounts
- Employees
- Machines
- Service assets
- Price lists

Master data tables must include ownership scope and lifecycle status.

### Transaction Tables

Transactions include business documents and operational movements:

- Sales invoices
- Purchase orders
- Goods receipts
- Warehouse transfers
- Manufacturing orders
- Journal entries
- Leave requests
- Loan requests
- POS receipts
- Service tickets
- Rental contracts

Transaction tables must preserve history and support audit requirements.

### Ledger And Movement Tables

Ledgers and movement tables record facts:

- Stock movements
- Accounting ledger entries
- Attendance events
- Payroll calculations
- Approval decisions
- Workflow transitions

These tables should be append-focused wherever possible.

Ledger and movement tables must not be treated as editable document tables. Corrections must be represented through reversal, adjustment, or compensating entries with audit trail.

## Transaction Boundaries

Database transactions are required for business actions that create multiple dependent facts.

Examples:

- Posting an invoice and creating accounting ledger entries.
- Receiving goods and creating stock movements.
- Approving a request and updating workflow state.
- Assigning a document number and creating the document.
- Publishing payroll and creating payslip records.

Rules:

- A transaction must include the authoritative document update, related ledgers or movements, workflow transition, and audit event when those facts represent one business action.
- External notifications, exports, and print rendering should normally happen after commit through events or background jobs.
- Retried commands must be idempotent where network retries or double submission are possible.
- Concurrency must be protected with database constraints, row versions, or locks where appropriate.

## Identifier Strategy

Use UUIDs for primary keys unless a specific performance or integration requirement justifies another choice.

Rules:

- Primary keys are technical identifiers.
- Business numbers are separate from primary keys.
- External identifiers must be stored separately and namespaced by integration.
- Public URLs must not depend on predictable numeric IDs.

## Document Numbering Strategy

Business documents need human-readable numbers such as invoice numbers, purchase order numbers, transfer numbers, and leave request numbers.

Numbering must be handled by a platform numbering service.

Numbering dimensions may include:

- Tenant
- Company
- Branch
- Fiscal year
- Module
- Document type
- Prefix
- Sequence

Rules:

- Document numbers must not be generated in React components.
- Document numbers must not be generated by counting rows in application code.
- Number assignment must be transaction-safe.
- Draft documents may use temporary numbers only if the module explicitly supports them.
- Posted or legally significant documents must not have their numbers reused.
- Voided or cancelled numbers must remain traceable if the document reached an official numbering stage.
- Numbering sequences must be protected from race conditions by database transactions or locks.
- Numbering gaps are acceptable only when the business and legal policy allows them; gapless numbering must be explicitly designed and tested.

## Status Strategy

Statuses must be module-owned and workflow-aware.

Rules:

- Do not use random text columns for uncontrolled statuses.
- Use typed status values backed by module registries.
- Database constraints should protect valid status values where practical.
- Application services must enforce valid transitions.
- RLS is not a workflow engine and must not replace status transition checks.
- Terminal statuses must be defined explicitly.
- Status changes must be recorded as workflow transition history for auditable documents.

## Audit Strategy

Nexora must maintain audit trails for sensitive operations.

Audit events should capture:

- Tenant ID
- Company ID where applicable
- Branch ID where applicable
- Actor user ID
- Actor role or permission context where useful
- Module
- Entity type
- Entity ID
- Action
- Previous values when appropriate
- New values when appropriate
- Reason or note when required
- Request metadata
- Timestamp

Audit logs must be append-only for normal application flows.

Auditing is required for:

- Permission changes
- Role changes
- User access changes
- Financial posting
- Inventory movements
- Payroll changes
- Approval decisions
- Workflow transitions
- Document cancellation
- Sensitive HR changes
- AI-executed actions

Audit tables must be designed for growth. High-volume audit data may require date-based partitioning, archival policies, or separate reporting access paths.

Audit records must be tamper-resistant in normal application flows:

- No update path for audit records except controlled retention metadata if required.
- No delete path except controlled retention procedures.
- Service-role maintenance must be logged separately.
- Audit reads must be permission-scoped.

## File Attachment Strategy

Files should be stored in Supabase Storage or a compatible object storage provider.

Database records must store metadata:

- Tenant ID
- Company ID where applicable
- Branch ID where applicable
- Owning module
- Owning entity type
- Owning entity ID
- File path
- File name
- MIME type
- File size
- Uploaded by
- Created at
- Security classification

Rules:

- File access must be permission-checked.
- Storage paths must include tenant-safe partitioning.
- Sensitive HR and financial documents must not be publicly accessible.
- Attachment deletion must respect business retention rules.
- File metadata must outlive object deletion where audit or retention policy requires a record of the attachment.
- Virus scanning or content validation should be introduced before accepting untrusted external uploads in production workflows.

## Indexing Strategy

Indexes must be designed for actual queries.

Default index considerations:

- `tenant_id`
- `company_id`
- `branch_id`
- Document date
- Document number
- Status
- Foreign keys
- Created timestamp
- Posted timestamp
- Product and warehouse keys for inventory
- Customer and supplier keys for commercial documents

Composite indexes should match common filters such as:

- `(tenant_id, company_id, document_date)`
- `(tenant_id, company_id, status)`
- `(tenant_id, branch_id, created_at)`
- `(tenant_id, product_id, warehouse_id)`

Indexes must be reviewed for high-volume tables before release.

Index review must include:

- Query plans for common list, search, report, and RLS-filtered queries.
- Composite indexes ordered by tenant scope first, followed by the most selective business filters.
- Partial indexes for active/open records when tables contain many historical rows.
- Unique indexes for business keys with correct tenant/company/branch scope.
- Foreign key indexes for joins and deletes.
- Separate indexes for full-text or trigram search when introduced.

Avoid indexing every column. Excess indexes slow writes on transaction-heavy tables such as invoices, stock movements, attendance events, and audit logs.

## Partitioning And Archival Strategy

Partitioning is not required on day one, but table designs must not block future partitioning.

Partitioning candidates:

- Audit logs.
- Notifications.
- Attendance events.
- Stock movements.
- Accounting ledger entries.
- Report result records.
- Integration logs.

Archival strategy must preserve legal and reporting requirements. Historical data may move to archive partitions or summary tables, but posted financial, payroll, and inventory facts must remain explainable.

## Reporting Data Strategy

Reports must not be built by loading large raw datasets into React.

Reporting options:

- Optimized SQL views for moderate reports.
- Materialized views for expensive aggregate reports.
- Precomputed summary tables for dashboards.
- Read replicas if available and needed later.
- Background report generation for very heavy outputs.

Reports must respect tenant, company, branch, and permission scope.

Report data models must distinguish:

- Operational read models for screens.
- Analytical summaries for dashboards.
- Export result sets for large downloads.
- Official financial reports that require reproducible calculations.

Financial and inventory reports must be reproducible for a period close. If master data changes later, closed-period reports must still be explainable.

## Printing Data Strategy

Printing must use stable print snapshots for documents where legal or business accuracy matters.

Examples:

- Sales invoices
- Purchase orders
- Delivery notes
- Payslips
- Warehouse transfers
- Service reports
- Rental contracts

Rules:

- Printed documents should not change unexpectedly because master data changed later.
- Print templates must be versioned when business format matters.
- Heavy print rendering should be moved to background or server-side rendering flows.
- Print snapshots must capture the exact values shown on official documents, including names, addresses, tax numbers, prices, discounts, totals, currency, locale, and template version.
- Print templates must not query arbitrary live data at render time for posted documents unless the module explicitly permits live rendering.

## Migration Strategy

Database changes must be managed through migrations.

Rules:

- No manual production schema edits.
- Every schema change must be reviewable.
- Destructive migrations require explicit backup and rollback planning.
- RLS policies must be migrated with tables.
- Indexes must be added with production data volume in mind.
- Seed data must not hardcode tenant-specific production IDs.
- Migrations that add non-null tenant, company, or branch columns must include a backfill plan.
- Migrations that change RLS policies must include access regression tests.
- Long-running index creation on production-sized tables must use safe deployment practices.

## Backup And Recovery Strategy

Backups are part of the architecture.

Requirements:

- Point-in-time recovery must be understood before production launch.
- Restore procedures must be tested, not only configured.
- Tenant-level data export and recovery requirements must be defined before enterprise customers require them.
- RLS, policies, storage metadata, and files must be included in recovery planning.

## Data Retention

Retention policies must be defined by module and jurisdiction.

Examples:

- Financial records may require long retention.
- Payroll and HR records may require restricted retention and access.
- Audit logs should be retained according to compliance requirements.
- Temporary files should have cleanup rules.

Data deletion must not break historical reports, posted documents, or audit trails.
